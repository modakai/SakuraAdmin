## Context

项目由 Spring Boot 3 后端和 Vue 3/shadcn-vue 前端组成，后端已有 `agreement`、`dict`、`user` 等按领域划分的模块，前端已有 `services/api`、`services/types`、`pages`、`components` 等分层。系统通知与公告会同时影响管理端发布流程、用户端展示流程、未读状态统计和数据持久化，因此需要在实现前明确数据模型、接口边界和展示规则。

通知消息与公告都属于平台向用户传递信息的能力，但使用场景不同：通知消息强调用户可查询、可标记已读，也可由系统事件根据模板自动生成；公告强调在有效期内主动展示、可置顶或弹窗。两者应共享基础内容模型和发布生命周期，同时在展示策略、阅读状态、接收端范围和模板触发上保留差异。

## Goals / Non-Goals

**Goals:**

- 建立统一的系统通知领域模块，覆盖通知消息与公告两类内容。
- 支持管理员管理草稿、发布、撤回、归档，并按接收端范围和目标范围投放内容。
- 支持管理员配置消息模板，让禁用用户等系统事件可以自动生成带动态参数的通知消息。
- 支持用户查看通知列表、通知详情、未读计数，并标记单条或全部通知为已读。
- 支持系统后台用户和用户端用户分别接收属于自身端的通知。
- 支持公告按生效时间窗口展示，过期后不再主动展示。
- 保持后端权限校验、数据过滤和前端状态查询边界清晰。

**Non-Goals:**

- 不实现 WebSocket、SSE、短信、邮件、微信公众号模板消息等实时或外部推送通道。
- 不实现复杂人群圈选、组织架构标签、A/B 投放或运营自动化。
- 不实现模板脚本执行能力；模板仅支持受控占位符替换。
- 不实现富文本编辑器以外的新内容编辑器能力。
- 不改变现有外观偏好 capability 的行为。

## Decisions

### Decision 1: 使用独立 `notification` 领域模块

新增后端包建议使用 `com.sakura.boot_init.notification`，内部继续按现有风格拆分 `controller/admin`、`controller/app` 或 `publicapi`、`model/dto`、`model/entity`、`model/vo`、`repository`、`service`、`service/impl`。这样可以避免把通知逻辑塞入 `support/system`，同时与现有业务模块结构保持一致。

备选方案是放入 `support/system`。该方案短期文件更少，但通知包含投放、已读、公告展示等业务状态，长期会让系统配置和业务消息耦合，不利于测试与维护。

### Decision 2: 通知与公告共享主表，已读状态独立存储

建议新增通知内容主表和阅读状态表。主表保存类型、标题、摘要、内容、级别、发布状态、接收端范围、目标范围、发布时间、生效时间、失效时间、置顶、弹窗等字段；阅读状态表按通知 ID、用户端类型和用户 ID 保存已读时间。未读计数通过“已发布且命中当前端、当前用户且未读”的条件计算。

备选方案是通知、公告拆成两张内容表。该方案字段更直接，但发布生命周期、目标范围、审计字段高度重复，会增加前后端接口和管理页面重复度。

### Decision 3: 接收端范围和目标范围分层建模

接收端范围使用枚举表达：`ADMIN`、`APP`、`ALL`，分别表示系统后台用户、用户端用户和全部用户。目标范围使用枚举表达：`ALL`、`ROLE`、`USER`，表示在接收端范围内继续按全部、角色或指定用户投放。角色或用户目标可以通过关联表或 JSON 字段保存，具体实现应优先选择当前项目已有的数据访问习惯。服务层必须统一封装“当前访问端和当前用户是否命中通知范围”的判断，管理端列表不依赖前端过滤。

备选方案是只保留一个目标范围字段，同时混合表达后台用户、用户端用户、角色和指定用户。该方案字段更少，但语义容易冲突，例如“全部用户”无法明确表示全部后台用户、全部用户端用户还是两端全部用户。

### Decision 4: 消息模板独立管理，系统事件只调用模板渲染服务

新增消息模板模型，保存模板编码、适用事件、标题模板、内容模板、启用状态、变量定义和默认接收端范围。模板内容支持 `{reason}` 这类占位符，系统事件触发时传入参数 Map，由服务层进行白名单变量校验和文本替换，再生成普通通知消息记录。禁用用户场景应传入禁用原因、操作人、用户 ID 等上下文，但最终入库的是渲染后的通知标题和内容。

备选方案是在禁用用户逻辑中硬编码通知文案。该方案实现最快，但管理员无法调整文案，也无法复用到其他自动通知场景。

### Decision 5: 前端区分管理端配置页、后台通知入口与用户端消息入口

管理端新增通知公告管理页面，负责列表筛选、表单编辑、发布撤回、归档、接收端范围和目标范围配置。管理端还需要新增消息模板配置页面，负责维护事件模板和变量说明。后台全局布局新增后台通知入口，用户端或全局布局新增用户端通知入口，分别显示各自端的未读计数、通知列表、详情和公告弹窗。API 客户端放入 `src/services/api/notification.api.ts`，类型放入 `src/services/types/notification.type.ts`，保持与现有服务分层一致。

备选方案是只做管理端页面，不做用户入口。该方案无法闭环验证通知到达、已读和公告展示，不满足系统通知能力的核心目标。

## Risks / Trade-offs

- [接收端范围和目标范围混淆] → 在数据模型和 DTO 中分别命名接收端范围与目标范围，并在服务层集中实现端类型、角色、用户三层匹配。
- [模板变量缺失或拼写错误] → 模板保存时校验变量声明，渲染时校验必填变量；缺失变量时阻止发送并记录业务错误。
- [自动通知误发] → 自动发送接口必须显式传入事件编码和目标用户，服务层按启用模板、接收端范围和目标范围统一校验后再创建通知。
- [未读计数性能下降] → 对通知状态、生效时间、类型、用户已读记录建立必要索引；分页查询时只返回当前页已读状态。
- [公告弹窗重复打扰用户] → 对公告阅读或关闭状态进行记录，已读或已关闭的公告不再重复弹出，除非公告重新发布生成新的展示版本。
- [撤回后仍被前端缓存展示] → 前端查询使用服务端状态为准；撤回接口成功后主动刷新通知列表、未读计数和公告查询缓存。
- [内容安全和权限越权] → 管理端接口必须要求管理员权限，用户端接口只能返回命中当前登录用户的已发布内容。

## Migration Plan

1. 新增通知、通知目标、通知阅读状态和消息模板相关数据表及必要索引，初始部署不迁移历史数据。
2. 后端先提供管理端通知 CRUD、模板 CRUD 与发布生命周期接口，再提供后台端和用户端查询、已读和公告接口。
3. 接入禁用用户等系统事件的模板通知调用，确保模板未启用或变量缺失时不会影响主业务事务。
4. 前端接入管理端通知页面、模板页面、后台通知入口和用户端入口，使用查询缓存刷新未读计数。
5. 若需要回滚，保留表结构但从菜单和路由隐藏入口，后端接口可通过权限或路由配置关闭访问。

## Open Questions

- 是否需要通知内容支持附件或跳转链接；当前设计仅保留可选链接字段，不实现附件管理。
- 角色标识应使用现有角色模型中的 ID 还是编码；实现时需以当前用户权限模型为准。
- 公告弹窗是否需要按设备维度记录关闭状态；当前默认按用户维度记录。
- 禁用用户事件的模板变量全集需要在实现时结合现有用户禁用接口确认，当前至少要求支持 `{reason}`。
