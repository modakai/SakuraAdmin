## Context

当前管理端已经存在多处外观相关能力：`theme.ts` 保存主题色、圆角和内容布局，`sidebar-config.ts` 保存菜单风格，`custom-theme` 弹窗提供快捷切换，`settings/appearance` 页面通过 `useSystemConfig` 尝试走系统配置接口。后端尚无对应系统配置模块，且用户已明确这些外观、主题、配置应按用户区分，并可存储在浏览器本地。

本设计将外观配置定位为“管理端用户本地偏好”。它不代表平台全局配置，也不需要后端持久化。实现重点是统一偏好模型、集中应用 DOM/CSS 状态、保留刷新后的本地恢复能力，并避免影响用户端页面。

## Goals / Non-Goals

**Goals:**

- 管理端提供一个清晰的外观偏好配置入口，覆盖当前主题能力并扩展更多 UI 偏好。
- 偏好按当前浏览器本地持久化，刷新页面和重新进入管理端后自动恢复。
- 统一现有主题、圆角、内容布局和菜单风格的状态来源，减少重复配置逻辑。
- 偏好只作用于管理端布局、管理端页面和管理端组件。
- 配置项保持枚举化和可校验，避免用户输入任意 CSS 造成样式失控。

**Non-Goals:**

- 不新增后端接口、数据库表或系统配置服务。
- 不做跨设备同步、账号云端同步或组织级统一下发。
- 不支持用户端外观配置。
- 不开放任意 CSS 变量编辑、主题包导入或自定义脚本。
- 不重构无关业务页面。

## Decisions

1. **使用管理端本地偏好 store 作为唯一状态源**

   现有 `theme.ts` 和 `sidebar-config.ts` 都使用 Pinia 持久化，但配置分散。实现时应收敛为一个管理端外观偏好模型，或至少让现有 store 共享同一套类型和初始化流程。推荐新增或重命名为 `admin-appearance-preferences`，包含主题色、明暗模式、圆角、字体、界面密度、内容布局、菜单风格、侧边栏默认状态、动效、表格斑马纹、面包屑和页面标题显示。

   备选方案是继续保留多个 store。它改动更小，但后续配置页需要协调多个状态源，测试和恢复逻辑更分散。

2. **通过集中应用器同步 DOM 和 CSS 变量**

   主题色、圆角、明暗模式等偏好最终要体现到 `document.documentElement` 的 class 和 CSS 变量上。实现时应提供一个集中 composable，例如 `useApplyAdminAppearancePreferences`，监听偏好变化并负责添加/移除 `theme-*` class、设置 `--radius`、切换 `.dark`、写入字体和密度相关 class。

   备选方案是继续在各个小组件内 `watchEffect`。它短期可行，但容易出现页面未挂载弹窗时偏好没有被应用的问题。

3. **配置项采用预设枚举，不开放任意值**

   主题色沿用现有 `THEMES`，圆角沿用 `RADIUS`。新增配置也采用固定枚举：字体、密度、动效、侧边栏状态等均由前端常量定义。这样可以保证表单校验、UI 预览和样式应用稳定。

   备选方案是允许用户填写颜色值和 CSS 变量。它更灵活，但会放大无效值、可读性和可访问性风险，不适合第一期。

4. **设置页作为完整配置入口，弹窗作为快捷入口**

   `settings/appearance` 或新的管理端外观配置页应承载完整配置项、重置默认值和预览说明。右上角 `custom-theme` 弹窗保留为快捷切换，只展示高频项，如明暗模式、主题色、圆角、内容布局和菜单风格。

   备选方案是只扩展弹窗。它入口轻，但配置项增多后弹窗会拥挤，不适合承载完整管理体验。

5. **移除外观配置对 `useSystemConfig` 的依赖**

   外观偏好既然不走后端，就不应再使用 `/system/config/*` 查询和写入。设置页应直接读写 Pinia 持久化 store。现有 `system-config.api.ts` 可暂时保留给其他潜在用途，但外观配置不再依赖它。

   备选方案是保留接口并在失败时回退本地。它会让需求语义回到“系统配置”，也引入不必要的错误状态。

## Risks / Trade-offs

- 本地存储无法跨浏览器和跨设备同步 → 在文案和需求中明确这是当前浏览器偏好。
- 偏好只在管理端应用，若路由布局边界不清可能泄露到用户端 → 应在管理端布局初始化应用器，并在非管理端页面避免读取管理端偏好。
- 配置项增多可能让设置页变复杂 → 使用分组呈现：主题、布局、导航、显示效果。
- 统一 store 可能影响现有 `custom-theme` 组件 → 通过兼容类型和小步迁移降低风险。
- 字体和密度配置需要全局 class 支撑 → 第一期开启有限预设，避免逐组件大规模重写。
