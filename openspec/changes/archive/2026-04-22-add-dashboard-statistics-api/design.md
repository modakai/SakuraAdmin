## Context

管理端 Dashboard 当前以静态数据为主，不能代表用户、通知、登录和审计日志的真实状态。项目后端已经存在 `user`、`notification`、`auth`、`audit`、`observability` 等模块，前端也已有 `src/pages/dashboard` 页面和 `src/services/api` 接口封装目录。

这个变更横跨后端聚合接口与前端 Dashboard 渲染。严格来说，Dashboard 不是运维观测页：用户总数、今日新增、通知数量、操作日志数量、登录趋势、最近操作日志属于管理端业务概览；JVM、CPU、接口错误率、异常 IP 等仍属于 `observability-ops-panel`。

## Goals / Non-Goals

**Goals:**

- 提供一个管理员可访问的 Dashboard 聚合接口，返回首页卡片、登录趋势和最近操作日志所需数据。
- 统一后端统计口径，避免前端继续维护静态数字或自行拼装多个业务接口。
- 前端 Dashboard 使用接口数据渲染，并覆盖加载、空数据和失败状态。
- 复用已有模块公开查询能力，避免 Dashboard 服务直接依赖其他模块内部实现。

**Non-Goals:**

- 不重做运维观测页，不把业务首页指标塞进 `/admin/observability/**`。
- 不新增面向普通用户或匿名用户的统计接口。
- 不在本次变更中设计复杂 BI、可配置报表、任意维度筛选或长周期数据仓库。
- 不要求小功能级别的临时前端测试沉淀为正式测试；如需验证，应遵守项目临时测试删除约束。

## Decisions

### Decision 1: 新增独立 Dashboard 聚合接口

新增管理员接口，建议路径使用 `/admin/dashboard/statistics` 或项目现有后台 API 命名风格下的等价路径。接口一次返回：

- `summary`: 用户总数、今日新增用户、通知数量、操作日志数量。
- `loginTrend`: 按日或按小时聚合的登录趋势点。
- `recentOperations`: 最近操作日志列表，包含操作人、动作、对象、结果、时间、IP 等展示字段。
- `sampleTime`: 后端采样时间，供前端显示数据新鲜度或调试。

选择聚合接口而不是让前端并发调用多个业务接口，是因为首页需要稳定、低耦合的数据契约。前端并发拼装会把统计口径、异常降级和模块依赖泄漏到 UI 层，后续维护成本更高。

### Decision 2: Dashboard 服务只依赖公开查询能力

后端实现应新增 Dashboard Controller/Service/DTO，并通过各模块的公开 Service、Repository 查询方法或 named interface 读取数据。若某个模块尚无合适的公开查询能力，应补充窄口径查询方法，而不是直接跨模块访问内部实体细节。

这个选择会多一些接口建模工作，但能守住模块边界。反过来，如果 Dashboard 直接读取所有表或内部实体，短期写得快，长期会让首页成为隐性耦合中心。

### Decision 3: 后端负责时间口径和脱敏

今日新增以服务器业务时区的自然日为统计口径；登录趋势默认返回最近 7 天或项目约定的首页周期；最近操作日志只返回展示字段，并屏蔽 token、password、authorization、captcha、secret、key 等敏感值。

时间口径不能让前端猜。否则同一数据在不同时区、刷新时机或部署环境下会出现难以解释的偏差。

### Decision 4: 前端保留现有 Dashboard 结构但替换数据来源

前端新增 `dashboard.api.ts` 和类型定义，Dashboard 页面在加载时请求聚合接口，将返回数据映射到现有概览卡片、趋势图和最近操作区域。现有静态 mock 数据只能作为加载骨架或空状态占位的视觉参考，不能作为成功态数据来源。

这个选择能控制改动范围。真正需要挑战的是“看起来有数据就够了”的假设：管理端首页如果显示的是假数据，比没有首页更危险。

## Risks / Trade-offs

- [Risk] 统计口径与业务预期不一致 → 在 DTO 与 spec 中明确今日、趋势周期、数量含义，并在接口响应包含采样时间。
- [Risk] 聚合查询影响首页加载性能 → 单接口内部执行有限数量的 count/list 查询，登录趋势使用一次范围查询聚合，必要时为登录时间、创建时间、审计时间补索引或分页限制。
- [Risk] 某些数据源尚未完备，例如登录成功记录缺字段 → 先按现有可用数据实现，缺口通过窄口径查询或轻量记录补齐，不引入大型报表系统。
- [Risk] 操作日志泄露敏感信息 → 最近操作日志仅返回展示 VO，后端统一脱敏，前端不展示原始请求参数。
- [Risk] Dashboard 与 observability 边界模糊 → 规格中明确业务首页统计不归入运维监控页面，运维模块继续聚焦系统状态、API 性能和安全事件。

## Migration Plan

1. 后端新增 Dashboard 统计 DTO、Service、Controller，并接入管理员权限控制。
2. 为用户、通知、登录、审计日志补充必要的公开查询方法。
3. 前端新增 Dashboard API 封装和类型定义，替换页面静态数据来源。
4. 验证接口成功、空数据、接口失败与无权限场景。
5. 若上线后发现统计查询压力过高，再引入轻量缓存或定时聚合；初版不预设复杂缓存，避免过早优化。

## Open Questions

- 通知数量应统计全部通知、未读通知，还是管理员可见通知？建议初版在接口字段名中明确为 `notificationCount`，实现时选择后台首页最有操作价值的口径，并在代码注释中写明。
- 后台上传直连 OSS 且没有本地文件记录，因此 Dashboard 不展示文件上传量，改为展示管理员操作日志数量。
- 登录趋势是否只统计成功登录？建议初版统计成功登录；失败登录继续留在观测/安全事件能力中展示。
