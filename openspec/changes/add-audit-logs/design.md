## Context

项目由 Spring Boot 3 后端和 Vue 3/shadcn-vue 前端组成，后端已有按业务领域划分的 `agreement`、`dict`、`notification`、`user` 等模块，也已有 `support` 包承载通用注解、AOP、拦截器、异常处理和系统级工具。审计日志属于横切能力：登录流程、管理员接口、异常处理、权限上下文、导出能力和前端后台页面都会受影响，因此需要明确采集边界、数据模型、脱敏策略和查询导出契约。

审计日志需要记录两类核心事件：登录日志用于安全追踪和登录失败诊断；管理员操作日志用于记录后台用户对业务资源的增删改查和配置变更。通用请求字段应尽量统一，操作描述通过注解补充业务语义，避免只看到 URL 而无法理解“删除用户”“修改系统配置”等关键动作。

## Goals / Non-Goals

**Goals:**

- 建立统一的审计日志领域模块，覆盖登录日志和管理员操作日志。
- 记录请求路径、HTTP 方法、IP、用户 ID、耗时、结果、异常信息、状态码、请求摘要和响应摘要。
- 支持后端注解声明重要操作描述、业务模块和操作类型。
- 支持后台按日志类型、用户、账号、IP、路径、方法、结果、描述和时间范围查询。
- 支持后台按筛选条件导出审计日志，并对敏感字段做脱敏和数量限制。
- 确保日志记录失败不影响原业务请求结果。

**Non-Goals:**

- 不实现链路追踪平台、日志聚合平台或外部 SIEM 系统对接。
- 不保存完整请求体、完整响应体、密码、令牌、验证码、密钥等敏感信息。
- 不实现用户端自助查看审计日志。
- 不实现日志归档到对象存储或冷热分层；仅预留保留策略配置边界。
- 不改变现有登录和后台业务接口的响应格式。

## Decisions

### Decision 1: 新增独立 `audit` 领域模块承载持久化和查询

后端新增 `com.sakura.boot_init.audit` 包，内部按现有风格拆分 `controller`、`service`、`repository`、`model/entity`、`model/dto`、`model/vo` 和 `enums`。通用注解和切面可放入 `support.annotation` 与 `support.aop`，但审计日志实体、查询、导出和业务服务放入独立领域模块，避免把数据访问和后台页面契约混入 `support`。

备选方案是全部放入 `support/system`。该方案文件少，但审计日志有独立表、筛选、导出和权限边界，长期放在 support 会让横切基础设施和业务查询能力耦合。

### Decision 2: 登录日志和操作日志使用统一主表，按日志类型区分

建议新增统一审计日志表，使用 `log_type` 区分 `LOGIN`、`ADMIN_OPERATION` 等类型，并保存路径、方法、IP、用户 ID、账号、耗时、结果、异常、操作描述、业务模块、操作类型、状态码、请求摘要、响应摘要、追踪 ID 和审计时间。登录日志没有路径或操作描述时允许为空，管理员操作日志应尽量完整记录请求和注解信息。

备选方案是登录日志与操作日志拆成两张表。拆表字段更直观，但后台查询、导出、索引、权限和展示组件会重复；统一表更适合“审计中心”入口，并保留通过类型筛选的清晰语义。

### Decision 3: 管理员操作使用注解 + AOP，登录日志使用认证流程显式调用

新增类似 `@AuditLog` 的注解，支持声明操作描述、业务模块、操作类型、是否记录请求摘要、是否记录响应摘要。AOP 在管理员接口执行前后采集耗时、用户上下文、请求元数据和异常信息。登录日志不依赖注解，而是在登录认证流程中显式调用审计服务，确保登录成功、账号不存在、密码错误、账号禁用等结果都能被准确记录。

备选方案是只用 Web 拦截器记录所有请求。拦截器覆盖面广，但缺少业务操作描述，也难以准确区分哪些后台操作是重要审计事件。注解让关键操作有明确语义，登录流程显式调用则保证认证失败也可记录。

### Decision 4: 异步写入审计日志，主请求只提交轻量事件

请求线程构造审计事件后交给审计服务异步持久化，落库失败只记录应用日志，不回滚或改变主业务结果。事件对象应在请求线程内完成必要上下文复制，避免异步线程访问失效的 request 或安全上下文。对异常信息、请求摘要和响应摘要做长度限制，避免大对象影响数据库和导出性能。

备选方案是同步写入。同步实现简单且更容易测试，但在高频后台操作或数据库波动时会增加接口耗时，并让审计能力影响主业务稳定性。

### Decision 5: 采集前统一脱敏，查询导出只返回已脱敏字段

新增脱敏规则，默认屏蔽 `password`、`token`、`authorization`、`secret`、`captcha`、`key` 等字段。请求摘要和响应摘要应在落库前完成脱敏和截断，后台查询详情与导出只读取已脱敏内容。异常信息只保存异常类型和短消息，堆栈仍留在应用日志中。

备选方案是完整保存后展示时脱敏。该方案便于排查，但风险更高，一旦数据库或导出文件泄露会暴露敏感信息；审计日志不应成为敏感数据二次存储点。

### Decision 6: 前端新增后台审计日志中心页面

前端新增审计日志 API 类型与客户端，例如 `src/services/api/audit-log.api.ts` 和 `src/services/types/audit-log.type.ts`。后台页面建议放在 `src/pages/audit-logs`，提供筛选表单、列表、详情抽屉或弹窗、结果状态展示和导出按钮，并接入现有菜单路由。导出按钮应使用当前筛选条件，并在导出中提示数量限制或后台返回的文件生成结果。

备选方案是只提供后端接口，不做前端页面。该方案无法满足后台查询、筛选、导出的闭环要求，也不利于管理员日常排查。

## Risks / Trade-offs

- [敏感信息泄露] → 落库前统一脱敏和截断，导出只使用已脱敏字段，默认不记录完整请求体和响应体。
- [日志记录影响主业务] → 采用异步持久化，失败仅记录应用日志，不改变原请求结果。
- [异步线程丢失上下文] → 在请求线程内复制用户、IP、路径、方法、注解元数据、耗时和异常摘要后再提交事件。
- [日志量增长导致查询变慢] → 对日志类型、用户 ID、账号、IP、路径、结果和审计时间建立索引，导出限制数量并要求时间范围。
- [注解覆盖不足] → 对登录日志显式记录，对未标注的重要管理员操作在任务中逐步补齐注解。
- [异常信息过长] → 只保存异常类型、业务错误码或短消息，并限制字段长度。
- [导出数据过大] → 导出接口必须复用筛选条件，并设置最大导出条数或异步任务边界。

## Migration Plan

1. 新增审计日志表、字段索引和必要枚举字典，初始部署不迁移历史应用日志。
2. 后端先实现审计服务、脱敏工具、异步写入和登录日志显式记录。
3. 再实现 `@AuditLog` 注解与管理员操作日志 AOP，并在用户管理、系统配置等关键接口补充注解。
4. 提供后台查询、详情和导出接口，接入审计查看权限。
5. 前端新增审计日志页面、API 客户端、类型定义、路由和菜单入口。
6. 若需要回滚，保留表结构但关闭菜单入口和审计 AOP 开关，登录流程保留不影响业务的失败降级。

## Open Questions

- 审计日志保留期限是否需要配置化；当前设计默认先落库并预留按时间清理任务。
- 导出格式使用 CSV 还是 Excel；实现时优先复用当前项目已有导出能力。
- 关键操作注解首批覆盖哪些接口；当前任务建议优先覆盖用户删除、用户禁用、角色/权限变更和系统配置修改。
- IP 获取是否需要兼容反向代理头；实现时需结合部署环境确认可信代理头策略。
